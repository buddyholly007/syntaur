## journal.rs (deploy journal append-only file)

```finding
id: journal-read-recent-memory-pressure
reviewer: gemini
file: syntaur-ship/src/journal.rs
line: 56
claim: read_recent uses fs::read_to_string to load entire monthly .jsonl files into memory; high-volume months could pressure memory.
verdict: TRUE
evidence: syntaur-ship/src/journal.rs:70 `let text = std::fs::read_to_string(&f)?;`
resolution: tracked — pre-existing. Real concern only at >>1k deploys/month with multi-KB failure traces; current cadence is <50/month so files stay tiny. rev_lines crate is the right fix when this matters.
```

```finding
id: journal-filter-map-silent-drop
reviewer: gemini
file: syntaur-ship/src/journal.rs
line: 73
claim: filter_map drops malformed lines silently; could hide corruption or write-bug regressions.
verdict: TRUE
evidence: syntaur-ship/src/journal.rs:73 `.filter_map(|l| serde_json::from_str(l).ok())`
resolution: tracked — pre-existing. Should warn-log on parse failure instead of dropping. Low priority since journal writers are local + single-process.
```

```finding
id: journal-write-non-atomic-pipe-buf
reviewer: gemini
file: syntaur-ship/src/journal.rs
line: 51
claim: O_APPEND + write_all is only atomic for writes < PIPE_BUF (~4KB); large failure_reason traces could interleave between concurrent writers across the NFS-shared vault.
verdict: TRUE
evidence: syntaur-ship/src/journal.rs:51 `f.write_all(line.as_bytes())?;`
resolution: tracked — pre-existing. Real concern across hosts (vault is NFS-shared between claudevm + gaming-pc). Mitigation: take an fs2 advisory lock per-append; or truncate failure_reason to <4KB before serialization. No corruption observed yet.
```
